\section{Intent and Verification Protocol}
\label{sec:ms-stab-protocol}

This chapter records the actual execution of the multiscale stabilization plan
after the publication-oriented redesign documented in
Chapter~\ref{ch:multiscale-publication-audit}.  The guiding rule for this
stabilization cycle was strict: neither the previous code, nor the existing
tests, nor the earlier documentation were accepted as ground truth without a
fresh audit against the implemented path.

The execution protocol used in this cycle was:
\begin{enumerate}
  \item audit the current normative path directly in code;
  \item reproduce uncertainty or weakness with a dedicated test or diagnostic;
  \item implement the correction or hardening;
  \item rerun a focused regression gate;
  \item document both the improvement and the residual debt in the same
    \LaTeX{} document.
\end{enumerate}

\section{Phase 1: Numerical Hardening of the Local Response}
\label{sec:ms-stab-phase1}

\subsection{Initial hypothesis}

The main numerical fragility of the public multiscale path was the local
tangent extraction: a one-sided perturbation with a fixed user-provided
\(h_{\mathrm{pert}}\), silent zero columns on failed perturbations, and hidden
post-processing by symmetry projection and diagonal clamping.

\subsection{Observed evidence before the change}

The audit of \texttt{NonlinearSubModelEvolver.hh} showed that the previous
implementation:
\begin{itemize}
  \item used only the \(+\) perturbation;
  \item used the same perturbation size for every generalized component;
  \item left the tangent column at zero if the perturbation solve diverged;
  \item restored displacement vectors but did not explicitly resynchronise the
    model state vector in the tangent loop;
  \item reported only the final tangent, thereby hiding which columns were
    valid and whether regularisation had actually modified the matrix.
\end{itemize}

\subsection{Implemented change}

The production operator remains the boundary-reaction operator, but the
stabilization no longer treats finite differences as the primary tangent path.
The current normative implementation first assembles the sparse local tangent
operator at the converged micro state, including the embedded-rebar penalty
terms, partitions it into free and constrained blocks, and condenses it
against the boundary kinematics.  With constrained increments
\(\delta \mathbf{u}_c = \mathbf{T}\,\delta \mathbf{q}\), the returned section
tangent is
\begin{equation}
  \mathbf{D}_{\mathrm{hom}}
  =
  \mathbf{T}^{\mathsf T}
  \mathbf{K}_{\mathrm{eff}}
  \mathbf{T},
  \qquad
  \mathbf{K}_{\mathrm{eff}}
  =
  \mathbf{K}_{cc}
  -
  \mathbf{K}_{cf}\mathbf{K}_{ff}^{-1}\mathbf{K}_{fc}.
\end{equation}
This is the first version of the multiscale operator in the repository that is
explicitly built from the linearised micro equilibrium equations rather than
from perturb-and-observe solves.  In the follow-up hardening pass, the
implementation was further upgraded from a dense prototype to a sparse solve of
the six required right-hand sides, which removed the need to materialise the
full dense Schur complement.

The present execution pass tightened that sparse route again by factoring the
Schur solve into a dedicated reusable utility and by adding a symbolic-pattern
workspace cache.  In practical terms, repeated calls to the local condensed
operator now reuse the symbolic factorisation pattern of the free block
\(\mathbf{K}_{ff}\) whenever the local topology is unchanged, while still
re-factorising the current numerical values at the converged state.  This is a
small but important architectural improvement: the code now makes the
performance boundary explicit instead of hiding symbolic-analysis cost inside
each operator evaluation.

Adaptive finite differences are retained only as a documented fallback.  When
the condensed operator cannot be formed---for example because a mixed element
does not expose explicit local tangent data or because the free block is
numerically singular---the implementation falls back to the adaptive scheme.
For each generalized component \(e_j\), the fallback uses
\begin{equation}
  h_j
  =
  \frac{\max\!\left(h_{\mathrm{pert}} L,\;
  10^{-3}\max(|q_j|, h_{\mathrm{pert}}L)\right)}{L},
\end{equation}
where \(L\) is the macro beam characteristic length and \(q_j\) is the
corresponding relative end degree of freedom.  The preferred tangent estimate
is
\begin{equation}
  \mathbf{D}_{\mathrm{hom}}^{(:,j)}
  \approx
  \frac{\mathbf{s}(\mathbf{e}+h_j \mathbf{e}_j)
       -\mathbf{s}(\mathbf{e}-h_j \mathbf{e}_j)}{2h_j},
\end{equation}
and the implementation falls back to one-sided differentiation only when one of
the two perturbation solves fails.

The present pass extended that fallback story into a runtime audit mode.  The
local solver can now request
\texttt{ValidateCondensationAgainstAdaptiveFiniteDifference}, which first
builds the sparse condensed tangent and then computes an adaptive
finite-difference reference on the same converged state.  What changed in this
execution pass is that the validation is no longer hard-wired to a single raw
Frobenius norm.  The code now exposes four norm families through
\texttt{TangentValidationNormKind}:
\begin{itemize}
  \item \texttt{RelativeFrobenius}, kept as the exact legacy metric;
  \item \texttt{StateWeightedFrobenius}, now the default publication-path
    acceptance norm;
  \item \texttt{DualEnergyScaled}, which combines the current generalized
    strain activity with the current generalized force activity;
  \item \texttt{SectionPowerScaledFrobenius}, a conservative audit norm that
    additionally scales moment rows into force-equivalent units.
\end{itemize}
The motivation is physical rather than cosmetic.  A raw Frobenius norm treats
all six tangent columns as equally relevant at the current state, even when
some generalized strain directions are nearly dormant.  The new state-weighted
norm therefore measures
\begin{equation}
  \eta_{\mathrm{sw}}
  =
  \frac{\lVert
      (\mathbf{D}_{\mathrm{cond}}-\mathbf{D}_{\mathrm{fd}})\mathbf{W}_q
    \rVert_F}
       {\max\!\left(
          1,
          \lVert \mathbf{D}_{\mathrm{cond}}\mathbf{W}_q \rVert_F,
          \lVert \mathbf{D}_{\mathrm{fd}}\mathbf{W}_q \rVert_F
       \right)},
\end{equation}
where \(\mathbf{W}_q\) is built from the currently active generalized strain
amplitudes.  The conservative audit norm adds a row scaling
\(\mathbf{W}_s\) and measures
\begin{equation}
  \eta_{\mathrm{de}}
  =
  \frac{\lVert
      \mathbf{W}_d(\mathbf{D}_{\mathrm{cond}}-\mathbf{D}_{\mathrm{fd}})
      \mathbf{W}_q
    \rVert_F}
       {\max\!\left(
          1,
          \lVert \mathbf{W}_d \mathbf{D}_{\mathrm{cond}}\mathbf{W}_q \rVert_F,
          \lVert \mathbf{W}_d \mathbf{D}_{\mathrm{fd}}\mathbf{W}_q \rVert_F
       \right)},
\end{equation}
where \(\mathbf{W}_d\) is built from the currently active generalized force
state and the same strain-activity proxy used by \(\mathbf{W}_q\).  The more
conservative section-power audit adds the geometry-based row scaling
\(\mathbf{W}_s\) and measures
\begin{equation}
  \eta_{\mathrm{sp}}
  =
  \frac{\lVert
      \mathbf{W}_s(\mathbf{D}_{\mathrm{cond}}-\mathbf{D}_{\mathrm{fd}})
      \mathbf{W}_q
    \rVert_F}
       {\max\!\left(
          1,
          \lVert \mathbf{W}_s \mathbf{D}_{\mathrm{cond}}\mathbf{W}_q \rVert_F,
          \lVert \mathbf{W}_s \mathbf{D}_{\mathrm{fd}}\mathbf{W}_q \rVert_F
       \right)}.
\end{equation}
The legacy metric
\begin{equation}
  \eta_{F}
  =
  \frac{\lVert \mathbf{D}_{\mathrm{cond}}
         - \mathbf{D}_{\mathrm{fd}} \rVert_F}
       {\max\!\left(1,
           \lVert \mathbf{D}_{\mathrm{cond}} \rVert_F,
           \lVert \mathbf{D}_{\mathrm{fd}} \rVert_F\right)}
\end{equation}
remains available for exact reproduction of previous audits.

On the mixed elastic reference state used by the new regression tests, the
measured metrics were
\begin{align}
  \eta_{F} &= 4.60221\times 10^{-2}, \\
  \eta_{\mathrm{sw}} &= 7.00812\times 10^{-3}, \\
  \eta_{\mathrm{de}} &= 7.00812\times 10^{-3}, \\
  \eta_{\mathrm{sp}} &= 6.94017\times 10^{-2}, \\
  \eta_{\max,\mathrm{sw}} &= 3.78672\times 10^{-1}, \\
  \eta_{\max,\mathrm{de}} &= 3.67884\times 10^{-1}, \\
  \eta_{\max,\mathrm{sp}} &= 7.36699\times 10^{-1}.
\end{align}
These numbers clarify the role of the new norms.  The state-weighted metric is
far smaller than the raw Frobenius gap because the discrepancy sits largely in
weakly active columns, the dual-energy metric preserves that global reduction
while slightly reducing the worst column mismatch on the audited state, and the
section-power-scaled metric is deliberately more conservative.

The public response object now records:
\begin{itemize}
  \item the tangent computation mode requested;
  \item the tangent scheme used;
  \item the status of the condensed operator attempt;
  \item the runtime validation status of the condensed operator;
  \item the selected validation norm kind;
  \item the configured validation tolerance;
  \item the measured global and per-column validation gaps;
  \item the row scales and column weights used by the selected norm;
  \item the sparse condensed solve residual when that path is attempted;
  \item the perturbation size used for each column;
  \item whether each column is valid;
  \item whether each column used the central stencil;
  \item the number of failed perturbation solves;
\item whether regularisation actually modified the returned tangent.
\end{itemize}

With a validation tolerance of \(7.5\times 10^{-2}\), the runtime audit keeps
the sparse condensed operator on the production path when
\texttt{StateWeightedFrobenius} is selected and reports
\texttt{TangentValidationStatus::Accepted}.  With a deliberately strict
tolerance of \(10^{-8}\) under \texttt{RelativeFrobenius}, the same case falls
back to adaptive finite differences and reports
\texttt{CondensedTangentStatus::ValidationRejected}.  That pair of tests closes
the observability gap around the tangent gate: the code now proves both the
accept and reject branches explicitly.

This pass also added a reproducible benchmark executable,
\texttt{fall\_n\_tangent\_validation\_benchmark\_test.exe}, to estimate the
expected production impact of the new default norm.  The benchmark evaluates
four mixed elastic states at tolerance \(4.5\times 10^{-2}\), measures a
direct condensed-only time \(t_{\mathrm{cond}}\), a direct forced-FD time
\(t_{\mathrm{fd}}\), and then defines the projected tangent-delivery cost
\begin{equation}
  T_{\mathrm{proj}}
  =
  \sum_i
  \begin{cases}
    t_{\mathrm{cond}}^{(i)}, & \text{if the audit accepts the condensed tangent},\\
    t_{\mathrm{fd}}^{(i)}, & \text{otherwise.}
  \end{cases}
\end{equation}
This is intentionally a projected cost, not the wall time of the validation
mode itself, because the validation mode still computes both operators by
construction.  On that benchmark family the measured results were
\begin{align}
  N_{\mathrm{accept},F} &= 0/4, &
  T_{\mathrm{proj},F} &= 8.64244~\mathrm{s}, \\
  N_{\mathrm{accept},\mathrm{sw}} &= 4/4, &
  T_{\mathrm{proj},\mathrm{sw}} &= 0.345097~\mathrm{s},
\end{align}
which corresponds to a projected speedup of \(25.04\times\) for the tangent
delivery stage of the outer FE\textsuperscript{2} loop on this benchmark.  The
benchmark is intentionally small, but it is fully reproducible and now part of
the multiscale stability gate.

\subsection{Resulting semantics}

The stabilization deliberately stopped calling every non-perfect response
``successful''.  The public status contract is now:
\begin{description}
  \item[\texttt{Ok}] the condensed operator was assembled successfully, all
    tangent columns are valid, no actual regularisation was applied, and any
    requested runtime validation either passed or was not requested;
  \item[\texttt{Degraded}] a usable response exists, but at least one
    fallback to finite differences, actual tangent regularisation, or
    unavailable runtime validation reference occurred;
  \item[\texttt{SolveFailed}] the response is not usable as a section operator;
  \item[\texttt{NotReady}/\texttt{InvalidOperator}] explicit non-solution
    states.
\end{description}

\subsection{Internal architectural hardening}

The numerical hardening was paired with an internal architectural split of the
local solver path.  Instead of leaving boundary conditions, persistent state
management, and boundary-reaction homogenisation interleaved inside
\texttt{NonlinearSubModelEvolver}, the cycle extracted three dedicated
components:
\begin{itemize}
  \item \texttt{PersistentLocalStateOps}: synchronisation, commit, and revert
    of the persistent continuum state;
  \item \texttt{LocalBoundaryConditionApplicator}: update of section-end
    kinematics and writing of the imposed boundary vector;
  \item \texttt{BoundaryReactionHomogenizer}: section forces from reactions,
    condensed tangent extraction, finite-difference fallback, diagnostics, and
    response classification.
\end{itemize}

This split was intentionally implemented as lightweight header-only helpers
composed through pointers and templates, so the hot path still avoids virtual
dispatch and heap churn.  Architecturally, this matters because it creates a
single explicit seam where the current sparse condensation / validation
fallback pair can later be upgraded again without changing the public
\texttt{SectionHomogenizedResponse} contract or the typed local model API.

The same cycle also extended \texttt{FEM\_Element} with the explicit local
linearisation API already present in \texttt{StructuralElement}.  This closes
an architectural gap in the mixed-element path: the condensed local operator
now sees the element DOF map and the element tangent even when the micro model
is stored behind \texttt{MultiElementPolicy} type erasure.

The same cycle also hardened the semantics of operator validity.  A local
response is now classified as \texttt{InvalidOperator} if any tangent column is
missing entirely.  One-sided fallback with a complete operator remains
\texttt{Degraded}; only a fully central, fully valid, non-regularised operator
is reported as \texttt{Ok}.  This change is small in code volume, but
scientifically important because it prevents an incomplete section operator
from being treated as merely ``low quality'' and then entering the FE\textsuperscript{2}
loop as if it were usable.

The subsequent hardening pass extended that same split to the observability
side of the local solver.  Two additional helpers were extracted from the
facade:
\begin{itemize}
  \item \texttt{LocalCrackDiagnostics}: owns the reduction from constitutive
    fracture fields to the public \texttt{CrackSummary} and the optional
    detailed \texttt{CrackRecord} cloud;
  \item \texttt{LocalVTKOutputWriter}: owns PVD bookkeeping and VTK emission
    for mesh, Gauss-point, crack-plane, and embedded-rebar outputs.
\end{itemize}
This matters architecturally for the same reason as the earlier split: the
local facade is now closer to a true coordinator of specialized components than
to a monolithic ``do everything'' object, while the hot numerical path still
stays template-based and free of virtual dispatch.

\subsection{Verification added in the current pass}

This execution pass added three focused checks on top of the previous
stabilization matrix:
\begin{itemize}
  \item \texttt{test\_sparse\_schur\_operator\_matches\_dense\_reference}:
    verifies that the reusable sparse Schur utility reproduces the dense
    reference contraction on a hand-auditable block system;
  \item \texttt{test\_sparse\_schur\_workspace\_reuses\_symbolic\_pattern}:
    verifies that repeated calls with unchanged sparsity reuse the symbolic
    factorisation pattern instead of analysing it again;
  \item \texttt{test\_condensed\_operator\_reuses\_symbolic\_pattern}:
    verifies the same reuse property through the full
    \texttt{NonlinearSubModelEvolver} path, not only in the low-level utility.
  \item \texttt{test\_validated\_condensed\_operator\_accepts\_elastic\_case}:
    verifies that the runtime audit mode keeps the sparse condensed tangent on
    the production path when the measured global operator gap stays below the
    configured tolerance.
  \item \texttt{test\_validated\_condensed\_operator\_rejects\_with\_strict\_tolerance}:
    verifies that the same runtime audit mode falls back explicitly to adaptive
    finite differences and reports \texttt{ValidationRejected} when the
    tolerance is made artificially strict.
\end{itemize}

The purpose of this extra layer is architectural as much as numerical.  The
code now proves, through tests, that the reusable sparse Schur module is not a
hidden refactor-only detail: its semantics propagate all the way to the public
\texttt{SectionHomogenizedResponse}.

After this pass, the full local release gate was rerun again through
\texttt{scripts/ci\_multiscale\_stabilization.ps1 -SkipConfigure}.  The gate
completed successfully, including the heavy multiscale example builds and the
\LaTeX{} document compilation, so the sparse Schur workspace refactor is not
only locally correct in isolation but also stable on the current release
surface.

\section{Phase 2: Protocol Hardening for Macro--Micro Iteration}
\label{sec:ms-stab-phase2}

\subsection{Initial hypothesis}

The checkpointed FE\textsuperscript{2} loop added in the redesign was close to
publication-ready, but the failure semantics were still too optimistic and too
opaque for a release gate.

\subsection{Errors confirmed during audit}

Two issues were confirmed in the code:
\begin{enumerate}
  \item on a macro trial-step failure inside
    \texttt{IteratedTwoWayFE2}, the macro checkpoint was not restored
    explicitly before returning;
  \item \texttt{CouplingIterationReport} did not expose the termination reason,
    rollback execution, regularisation presence, or failed coupling sites.
\end{enumerate}

\subsection{Implemented change}

The orchestrator now records:
\begin{itemize}
  \item \texttt{CouplingTerminationReason};
  \item whether rollback was performed;
  \item whether relaxation changed the current iterate;
  \item whether regularisation was detected among accepted local responses;
  \item the set of hard-failure sites;
  \item a count of regularised submodels.
\end{itemize}

Most importantly, the macro checkpoint is now restored explicitly on macro-step
failure before the FE\textsuperscript{2} step returns.  This closes a real
semantic gap that could otherwise leave a dirty trial state in memory after a
reported failure.

\subsection{Scientific root-cause analysis of the first cracked Case~5 point}
\label{sec:ms-stab-case5-root-cause}

The later cyclic-validation audit showed that the first cracked
FE\textsuperscript{2} point of Case~5 had not been failing for a single
reason.  Three coupled defects were involved:
\begin{enumerate}
  \item the local Ko--Bathe factory was constructing
    \texttt{KoBatheParameters} with positional arguments
    \texttt{(fc, 0.06, lb)}, which the parameter class interprets as
    \((f_c', t_p, G_f)\) rather than \((f_c', G_f, l_b)\);
  \item the inferred local crack-band length was using a cube-root-volume
    proxy instead of the article's local max-side interpretation from
    Fig.~7(b)~\cite{KoBathe2026};
  \item the local checkpoint/restore path only preserved displacement-like
    quantities and therefore disabled exact rollback of the constitutive state
    in trial-mode FE\textsuperscript{2} iterations.
\end{enumerate}

The first issue is a genuine scientific implementation bug.  The intended
parameter vector is
\begin{equation}
  (f_c', t_p, G_f, l_b),
\end{equation}
but the old factory call effectively set
\begin{equation}
  t_p = 0.06,
  \qquad
  G_f = l_b,
  \qquad
  l_b = 100~\text{mm},
\end{equation}
which is dimensionally inconsistent and materially alters the tension-softening
law
\begin{equation}
  \varepsilon_{tu} = \frac{2G_f}{f_t l_b}.
\end{equation}
That bug is now closed by an explicit named mapping through
\texttt{KoBatheConcreteMaterialFactory::parameters()} and guarded by dedicated
tests.

The second issue was not merely a documentation discrepancy.  Because
\(\varepsilon_{tu}\) scales inversely with \(l_b\), the choice of length scale
changes the onset and slope of the crack-softening branch directly.  The
current code now uses
\begin{equation}
  l_b^{\mathrm{local}}
  =
  10^3 \max(\Delta x,\Delta y,\Delta z)
\end{equation}
for the inferred local path, which matches the interpretation stated in
Section~2.1.5 of the article.

The third issue explains why the iterated FE\textsuperscript{2} path was much
more fragile than the one-way path.  In the one-way case, the local model
always advanced from the last committed state.  In the iterated case, each
trial outer iteration needed to:
\begin{enumerate}
  \item restore the exact micro constitutive history;
  \item solve a new local boundary-value problem;
  \item either accept and commit at convergence or roll back completely.
\end{enumerate}
Without a full model checkpoint, the local adaptive continuation logic was
operating on partially stale history data even when the displacement field had
been restored.

After fixing those three issues and tuning the exploratory short-run profile to
a 12-substep first ramp with larger SNES and adaptive budgets, the first
cracked Case~5 point
\((p = 0.0555556,\; d = 2.5~\mathrm{mm})\) converged with zero failed
sub-models and two outer FE\textsuperscript{2} iterations.  Therefore the
scientific frontier has moved: the first cracked point is no longer the
blocking barrier, and the open debt now lies in later amplitudes of the
\texttt{extended50} protocol.

\paragraph{Material tangent audit: explicit but not yet promotable.}
The next scientific question was whether the first-cracked Case~5 point could
be made even more robust by enabling the numerical consistent tangent of the
\texttt{KoBatheConcrete3D} relation inside the FE\textsuperscript{2}
sub-models.  The implementation now exposes that choice explicitly through the
sub-model concrete factory and through the cyclic-validation CLI flag
\texttt{--submodel-consistent-material-tangent}.  However, the audited
benchmark showed that this option is still experimental:
\begin{itemize}
  \item with the default fracture-sec tangent, the first cracked point
    converged in about \(771.31\)\,s and all four sub-models completed
    \texttt{FirstSolveRamp};
  \item with the consistent-tangent override, the wall time dropped to about
    \(595.14\)\,s but two sub-models failed at an achieved fraction of
    \(0.916667\), the multiscale iteration rolled back, and the step ended in
    \texttt{MicroSolveFailed}.  This is now a historical intermediate
    benchmark: after the later first-ramp and tail-continuation corrections,
    the default fracture-sec path moved the frontier beyond the local micro
    failure and into the macro FE\textsuperscript{2} re-solve.
\end{itemize}
This result is scientifically valuable precisely because it is negative:
``closer to the article'' is not yet synonymous with ``more robust'' in the
current 3D FE\textsuperscript{2} implementation.

\paragraph{Material-level implementation error closed in the same pass.}
The material audit also found a more concrete constitutive inconsistency than
the tangent-mode debate.  When the 3D plastic return mapping updated the
plastic strain within a step, the stored crack opening and crack-closure flag
were still reflecting the pre-return elastic strain.  The implementation now
refreshes those quantities from the final elastic strain state
\(\boldsymbol{\varepsilon}^{e}_{t+\Delta t}\), so the committed crack
kinematics are once again consistent with the committed plastic strain.  This
does \emph{not} yet make the whole 3D algorithm a line-by-line transcription of
the Ko--Bathe flow chart, but it removes a real constitutive-state bug that was
capable of polluting the FE\textsuperscript{2} local solve exactly at crack
onset.

\section{Phase 3: Physical Regression Matrix}
\label{sec:ms-stab-phase3}

\subsection{Added regression coverage}

The stabilization cycle introduced a small deterministic matrix of physical
reference cases using the real \texttt{NonlinearSubModelEvolver} path:
\begin{itemize}
  \item pure axial loading;
  \item pure bending about \(y\);
  \item pure bending about \(z\);
  \item pure shear along \(y\);
  \item pure shear along \(z\);
  \item pure torsion.
\end{itemize}

The tests check:
\begin{itemize}
  \item convergence of the local sub-model;
  \item availability of the homogenised response;
  \item positivity of the expected tangent diagonal;
  \item completion of the tangent diagnostics;
  \item use of \texttt{LinearizedCondensation} and the
    \texttt{forces\_consistent\_with\_tangent} flag in the elastic cases;
  \item dominance or mode-consistent prominence of the expected resultant;
  \item linearised consistency in the elastic axial case;
  \item boundary-reaction versus volume-average agreement in the elastic axial
    case;
  \item axial macro--micro power consistency.
\end{itemize}

The stabilization outcome is now much easier to interpret.  The legacy

\end{align}
which corresponds to a projected speedup of \(1.38\times\) for the

\(19.96\times\) for the dual-energy gate on the same benchmark.  The result is

the strain-only weighting is already a better physical proxy than pure
Frobenius, but on this particular four-state family it still rejects three
cases that the dual-energy metric accepts.  The word \emph{projected} matters
here: the validation mode still computes both the condensed and FD operators,
so the benchmark estimates the expected production benefit of the acceptance
criterion rather than claiming that the validation mode itself became faster by
that factor.

\subsection{Coupling residual norms and projected outer-iteration cost}

The next gap closed in this cycle was semantic rather than algebraic: the
operator-validation path had already adopted the weighted norm family, but the
outer FE\textsuperscript{2} convergence loop still consumed raw relative
residuals.  That meant the code could accept a condensed operator with a
state-weighted criterion while still forcing extra fixed-point iterations
because the outer tangent residual remained tied to a pure Frobenius norm.

The implementation now aligns those two layers.  \texttt{MultiscaleAnalysis}
stores separate norm selections for force and tangent residuals, both defaulted
to \texttt{StateWeightedFrobenius}.  For a local macro strain proxy
\(\mathbf{e}_{\mathrm{ref}}\), the force residual at one coupling site is now
measured as
\begin{equation}
  r_{f,\mathrm{sw}}
  =
  \frac{\lVert \mathbf{W}_q
    (\mathbf{s}_{\mathrm{macro}}-\mathbf{s}_{\mathrm{micro}})\rVert_2}
  {\max\!\left(
    1,
    \lVert \mathbf{W}_q \mathbf{s}_{\mathrm{macro}}\rVert_2,
    \lVert \mathbf{W}_q \mathbf{s}_{\mathrm{micro}}\rVert_2
  \right)},
\end{equation}
and the tangent residual between two fixed-point iterates is
\begin{equation}
  r_{D,\mathrm{sw}}
  =
  \frac{\lVert (\mathbf{D}^{(k)}-\mathbf{D}^{(k-1)})\mathbf{W}_q\rVert_F}
  {\max\!\left(
    1,
    \lVert \mathbf{D}^{(k)}\mathbf{W}_q\rVert_F,
    \lVert \mathbf{D}^{(k-1)}\mathbf{W}_q\rVert_F
  \right)}.
\end{equation}
The same code path can still be switched back to
\texttt{RelativeFrobenius} for exact historical reproduction or to
\texttt{SectionPowerScaledFrobenius} for conservative audit campaigns.  The
dual-energy family is also available and reuses the same generalized-force and
generalized-strain reference state employed by the condensed-operator audit.
The important part is that the selected norm family is now stored explicitly in
\texttt{CouplingIterationReport}, together with the effective component scales
for force residuals and the row/column scales used for tangent residuals.

To make the expected convergence effect reproducible, a second benchmark,
\texttt{fall\_n\_coupling\_residual\_benchmark\_test.exe}, was added.  It uses
the same four mixed elastic states as the tangent-validation benchmark, forms a
surrogate fixed-point pair from the condensed and forced-FD tangents at each
state, and measures the local condensed-response cost that would be paid again
if the outer criterion rejected the iterate.  The measured results at
tangent tolerance \(4.5\times 10^{-2}\) were
\begin{align}
  N_{\mathrm{accept},F} &= 0/4, &
  T_{\mathrm{proj},F} &= 0.795333~\mathrm{s}, \\
  N_{\mathrm{accept},\mathrm{sw}} &= 4/4, &
  T_{\mathrm{proj},\mathrm{sw}} &= 0.397666~\mathrm{s}, \\
  N_{\mathrm{accept},\mathrm{de}} &= 4/4, &
  T_{\mathrm{proj},\mathrm{de}} &= 0.397666~\mathrm{s},
\end{align}
for a projected outer-iteration speedup of exactly \(2.0\times\) on this
surrogate benchmark.  Again, the word \emph{projected} is deliberate: the test
does not claim that the whole FE\textsuperscript{2} solve is twice as fast in
general, only that this norm change removes one otherwise spurious extra outer
iteration on the benchmarked mixed states.

\section{Post-Stabilization Merge Audit with \texttt{origin/master}}
\label{sec:ms-stab-merge-audit}

Once the stabilization gate was green locally, the branch was not merged
blindly with \texttt{origin/master}.  Instead, the integration was treated as
another audit phase: every conflicting file was reviewed under the same rule
used throughout this stabilization cycle, namely that older or semantically
weaker code is not accepted just because it came from the remote default
branch.

\subsection{Audit hypothesis}

The initial hypothesis was that the merge could reintroduce one of three
classes of defects:
\begin{enumerate}
  \item semantic regression, where the remote side reintroduced placeholder or
    legacy code that weakens the now-documented public API;
  \item architectural regression, where a low-level header would be coupled
    again to non-normative concerns;

    silently change the behaviour of multiscale builds.

\subsection{Implemented changes}

The follow-up changes touched three areas:
\begin{itemize}

    when the CSV files are present, it can produce not only the classical
    hysteresis overlays but also recorder-based plots for global damage, crack
    evolution, and solver diagnostics.

\begin{itemize}

This observation is scientifically useful because it rules out a tempting but
incorrect diagnosis.  The one-way FE\textsuperscript{2} path is not currently
limited by immediate operator unavailability at the first cracked point.  It is
limited by \emph{cost} and by a \emph{response mismatch}: the continuum
sub-models accumulate many cracks at the same global drift level where the
structural-only model still reports relatively modest macro damage.

\subsection{Executor caveat}

One explicit caveat emerged from the audit and is now part of the documented
truth baseline: the cyclic FE\textsuperscript{2} driver must not currently be
run with \texttt{OpenMPExecutor}.  In the audited environment, that path caused
PETSc wrong-state failures associated with interleaved norm begin/end calls.
The code now therefore hardwires the cyclic FE\textsuperscript{2} validation
driver to \texttt{SerialExecutor}.  This is not a theoretical objection to
threaded micro execution in general; it is a concrete statement that this
specific PETSc-backed path is not yet validated for threaded execution.

\subsection{Compile-time evidence}

The compile-time audit also remained deliberately empirical.  After the earlier
PCH enablement and the move to parallel per-target builds in the CI script and
workflow, the heavy validation target still showed large rebuild costs in the
current environment:
\begin{itemize}
  \item rebuild of \texttt{fall\_n\_table\_cyclic\_validation} after the
    follow-up instrumentation changes: approximately 889~s;
  \item rebuild after the protocol-preserving FE\textsuperscript{2} step-cap
    correction: approximately 947~s.
\end{itemize}

This is important because it keeps the compile-time narrative honest.  PCH and
parallel target builds improved the gate and reduced redundant work, but they
did \emph{not} solve the underlying translation-unit size problem of
\texttt{main\_table\_cyclic\_validation.cpp}.  The next credible compile-time
improvement remains the one already identified during the audit: split this
driver into multiple translation units so that rebuilds of the FE\textsuperscript{2}
cases do not force recompilation of the entire validation suite.

\subsection{Second pass: recorder semantics, driver split, and adaptive continuation}

The next execution pass addressed three coupled problems that had become clear
only after the first exploratory FE\textsuperscript{2} cyclic runs:
\begin{enumerate}
  \item the cyclic recorders were semantically dishonest for constitutive laws
    that do not expose a scalar damage channel;
  \item the validation executable still hid too much compile-time cost behind a
    monolithic entrypoint;
  \item the first load reversal and the first \(+5\)\,mm excursion required
    adaptive continuation in the local sub-models, and the existing adaptive
    solver still contained an exact-budget classification bug.
\end{enumerate}

\paragraph{Recorder semantic correction.}
The audit of \texttt{KoBatheConcrete3D}, \texttt{Material.hh},
\texttt{FEM\_Element.hh}, and the cyclic FE\textsuperscript{2} driver confirmed
that the earlier recorder path silently mapped ``no scalar damage field
available'' to zero.  The stabilized execution now keeps that distinction
explicit:
\begin{equation}
  \texttt{damage\_scalar\_available} = 0
  \quad\Longrightarrow\quad
  \texttt{peak\_submodel\_damage\_scalar} = \texttt{NaN}.
\end{equation}
For the present cyclic FE\textsuperscript{2} path, the trustworthy fracture
channels are therefore the crack counts, crack opening envelope, and the
octahedral invariants \(\sigma_{o,\min}\) and \(\tau_{o,\max}\).  This matters
scientifically because it prevents a heavily cracked sub-model from being
misread as an undamaged one purely because the constitutive snapshot did not
expose a scalar damage index.

\paragraph{Fourth pass: local dependency umbrella instead of \texttt{header\_files.hh}.}
The next compile-time-oriented pass targeted a different problem: even after
the TU split and the validation-specific PCH, the heavy cyclic-validation path
still entered the build through the repository-wide umbrella
\texttt{header\_files.hh}.  That was architecturally weak for two reasons:
\begin{enumerate}
  \item it forced the cyclic-validation support layer to depend on a
    near-global transitive include surface instead of a clearly delimited local
    dependency set;
  \item it made the target-specific PCH conceptually misleading, because the
    supposedly local PCH still started from the global umbrella.
\end{enumerate}

The implementation therefore introduced
\texttt{src/validation/TableCyclicValidationDeps.hh} as a local umbrella for
the cyclic-validation suite and changed
\texttt{TableCyclicValidationSupport.hh} to include that header instead of
\texttt{header\_files.hh}.  In the same pass,
\texttt{TableCyclicValidationPch.hh} was changed to include only the stable
driver API and the new local dependency umbrella, not the support header
itself.  The implementation criterion was explicit: edits to support logic
should no longer invalidate the PCH solely because the PCH re-included the
support header transitively.

The measured result of that pass is scientifically useful precisely because it
was mixed:
\begin{itemize}
  \item touching \texttt{main\_table\_cyclic\_validation.cpp} rebuilt in
    15.277~s, preserving fast CLI feedback;
  \item touching \texttt{TableCyclicValidationFE2.cpp} rebuilt in
    457.552~s in the audited environment.
\end{itemize}
This is not presented as an unconditional compile-time success.  The local
umbrella clearly improved dependency hygiene and made the build surface more
honest, but it did \emph{not} close the heavy-TU frontier.  In fact, on the
audited machine the latest FE\textsuperscript{2} TU measurement was slower than
the earlier 332.621~s best-case figure.  The correct conclusion is therefore
not that the refactor ``made compilation faster'', but that it removed a
global architectural smell while preserving fast feedback for the thin CLI.
The remaining compile-time debt still lives in the FE\textsuperscript{2}
translation unit itself, which continues to instantiate a very large fraction
of the multiscale and nonlinear local-solver surface.

The additional runtime-I/O extraction in the present pass is therefore best
understood as a \emph{cost-of-change} improvement rather than a promise of
immediate full-build speedup.  Recorder buffering, file rewrite logic, and
turning-point row bookkeeping are now isolated from the main FE\textsuperscript{2}
driver implementation, so edits in that slice no longer require reopening the
same large translation unit as the nonlinear multiscale runtime.

The present follow-up pass continued that direction by extracting the
sub-model material-factory decisions into
\texttt{TableCyclicValidationSubmodelFactories}\allowbreak\texttt{.hh/.cpp}.
This new split does not magically solve the heavy-TU problem either; it simply
moves another scientifically volatile slice (material profile selection,
default crack-band length inference, and tangent-mode wiring) out of the main
FE\textsuperscript{2} driver.  The measured incremental rebuilds on the local
audit were both about \(44\)\,s when touching either the new factory TU or the
main FE\textsuperscript{2} TU, which means the current benefit is again mainly
architectural honesty rather than a dramatic compile-time win.  That result is
kept intentionally explicit here so the manuscript does not over-claim the
effect of the split.

The next split in the same line of work moved the post-step crack-summary
aggregation and recorder-row assembly into
\texttt{TableCyclicValidationFE2StepPostprocess}\allowbreak\texttt{.hh/.cpp}.
This again should not be read as a miraculous full-build optimization.  The
measured benefit is mainly a reduction in \emph{cost of change}: edits in the
FE\textsuperscript{2} runtime loop no longer require reopening the recorder and
CSV assembly slice, while recorder changes no longer require recompiling the
same driver logic that owns the nonlinear step/restart flow.  The heavy
translation-unit frontier is therefore still present, but the architectural
boundary is now materially cleaner than before.

The current pass continued that split once more by extracting the turning-point
restart retuning logic into
\texttt{TableCyclicValidationFE2Restart}\allowbreak\texttt{.hh/.cpp}.  This is
again a \emph{cost-of-change} improvement more than a raw compile-time one:
the FE\textsuperscript{2} runtime loop now reads more clearly as
``advance / diagnose / optionally restart'', while the policy that tightens the
local submodel budgets on restart is isolated in a dedicated component.  The
heavy translation unit is therefore still expensive, but the protocol logic is
less entangled with restart bookkeeping than in the previous pass.

The follow-up pass documented in this chapter then extracted the recorder
bootstrap into
\texttt{TableCyclicValidationFE2Recorders}\allowbreak\texttt{.hh/.cpp}.  This
new slice owns the creation of recorder headers, the per-submodel crack-count
columns, the solver-diagnostics schema, and the initial hysteresis row.  The
scientific motivation is modest but important: the FE\textsuperscript{2}
runtime loop now focuses on protocol ownership and step execution, while the
CSV/recorder schema is isolated in a dedicated component that can evolve
without reopening the same translation unit as the nonlinear coupling logic.

The compile-time benchmark for this split was measured at object level with
\texttt{ccache} disabled.  The audited results were:
\begin{itemize}
  \item recompiling
    \texttt{CMakeFiles/fall\_n\_table\_cyclic\_validation.dir/src/validation/TableCyclicValidationFE2.cpp.obj}:
    177.194~s;
  \item recompiling
    \texttt{CMakeFiles/fall\_n\_table\_cyclic\_validation.dir/src/validation/TableCyclicValidationFE2Recorders.cpp.obj}:
    38.634~s.
\end{itemize}
This is the honest result that matters for day-to-day development: the split
does not remove the heavy FE\textsuperscript{2} compilation frontier, but it
does create a recorder/IO slice whose isolated rebuild cost is about
\(177.194/38.634 \approx 4.59\) times smaller than reopening the main
runtime TU.

\paragraph{Adaptive continuation in the cyclic FE\textsuperscript{2} path.}
The first exploratory one-way FE\textsuperscript{2} run with an
8-substep \texttt{FirstSolveRamp} reached \(+2.5\)\,mm but still failed at the
first reversal \(-2.5\)\,mm.  The reason was not the initial ramp; it was the
fact that the subsequent solve path still attempted a full post-cracking step
without adaptive continuation.  The driver was therefore extended with
explicit controls for:
\begin{itemize}
  \item enabling adaptive sub-stepping from the first post-ramp step;
  \item choosing the arc-length activation threshold;
  \item choosing the adaptive substep and bisection budgets.
\end{itemize}
With adaptive continuation enabled from the start, the one-way cyclic
FE\textsuperscript{2} path completed the first reversal and the return to zero.
The \(-2.5\)\,mm reversal required exactly two adaptive substeps in every
sub-model; the zero-return step also required two adaptive substeps in every
sub-model.

\paragraph{Exact-budget bug and scientific consequence.}
The first attempt at the \(+5\)\,mm excursion then exposed a more subtle issue:
one sub-model reached \(\texttt{achieved\_fraction}=1.0\) at the same time it
hit the hardcoded 30-substep budget, but the solver still returned failure
because the budget check was executed after the successful update and did not
distinguish ``budget exhausted before completion'' from ``budget exhausted
exactly at completion''.  That was a real correctness defect, not merely a
tuning issue.

The fix had two parts:
\begin{enumerate}
  \item classify the adaptive step as failed only when the budget is exhausted
    \emph{and} the progress fraction remains \(< 1\);
  \item expose the adaptive substep/bisection budgets as explicit evolver and
    CLI configuration rather than hidden constants.
\end{enumerate}

This correction was locked by a dedicated regression test,
\texttt{test\_adaptive\_substepping\_accepts\_exact\_budget\_completion},
which constructs a case that converges in exactly two adaptive substeps and
verifies that the solver remains classified as converged.

The \(+5\)\,mm point required substantial local continuation:

\paragraph{Coordinator lifetime defect after the setup split.}
The next audit step was not a constitutive improvement but a correctness
correction in the FE\textsuperscript{2} driver surface itself.  Once the cyclic
setup was split into \texttt{TableCyclicValidationFE2Setup}\allowbreak
\texttt{.hh/.cpp}, the setup function started returning local evolvers that
still held raw \texttt{MultiscaleSubModel*} pointers into a temporary
\texttt{MultiscaleCoordinator}.  When the function returned, the coordinator
was destroyed, but the local evolvers retained the dangling pointers.  The
first subsequent call to rebuild the boundary conditions then read corrupted
face-node caches and failed inside
\texttt{compute\_boundary\_displacements()}.

The backtrace was decisive: the crash no longer pointed to the multiscale
response container, but to a nonsensical
\texttt{std::vector<PetscInt>} size inside the face-node cache.  In the audited
debugging session, the first corrupted call attempted to reserve a vector with
an absurd boundary-node count, which made the issue clearly architectural
rather than material.

The correction had three parts:
\begin{enumerate}
  \item \texttt{FE2CaseContext} now owns the
    \texttt{MultiscaleCoordinator} explicitly, and the declaration order keeps
    the coordinator alive until after the multiscale analysis and its local
    evolvers are destroyed;
  \item \texttt{compute\_boundary\_displacements()} now checks that the cached
    face-node list does not exceed the owning domain node count and that every
    cached node id lies inside the domain;
  \item \texttt{LocalBoundaryConditionApplicator} now throws an explicit error
    if the face caches are empty, instead of silently proceeding into undefined
    behaviour.
\end{enumerate}

\paragraph{Explicit no-flow correction in the Ko--Bathe 3D path.}
Once the driver lifetime bug was removed, the material audit returned to the
foreground.  A direct comparison against Table~1 of
Ko--Bathe~\cite{KoBathe2026} showed that the 3D implementation was still
missing the explicit top-level branch split between
\emph{compressive loading} and \emph{no-flow}.  The code now classifies the
branch with the trial octahedral invariants,
\begin{align}
  \text{compressive flow}
  &\iff
  {}^{t+\Delta t}\sigma_o^{\mathrm{tr}} < 0
  \;\wedge\;
  {}^{t+\Delta t}\tau_o^{\mathrm{tr}} > {}^{t}\tau_o^{\max},
  \\
  \text{no-flow}
  &\iff
  {}^{t+\Delta t}\sigma_o^{\mathrm{tr}} \ge 0
  \;\vee\;
  {}^{t+\Delta t}\tau_o^{\mathrm{tr}} \le {}^{t}\tau_o^{\max},
\end{align}
and only the compressive-flow branch is now allowed to enter the
compression-coordinate return mapping.  Two new regressions,
\texttt{test\_unloading\_is\_classified\_as\_no\_flow\_3d} and
\texttt{test\_tension\_is\_classified\_as\_no\_flow\_3d}, verify that once a
cracked mixed-compression state unloads or returns to tension, the effective
plastic strains remain frozen.

The same audit then showed a second constitutive defect: the 3D no-flow branch
was still carrying the compressive fracture/plasticity tangent into tensile and
unloading states, and it still lacked the explicit coupling update of
Eq.~(26b).  The corrected branch now uses
\begin{equation}
  C_{ijkl}^{\mathrm{NF}}
  =
  \begin{cases}
    C_{ijkl}^{E}(K_e,G_e), & \text{tension}, \\
    C_{ijkl}^{E}(K_s,G_s), & \text{compressive unloading},
  \end{cases}
\end{equation}
followed by the tensorial update
\begin{equation}
  {}^{t+\Delta t}\!\varepsilon^{p}_{ij}
  =
  {}^{t}\!\varepsilon^{p}_{ij}
  -
  {}^{t+\Delta t}\!\varepsilon^{e}_{ij}(N)
  +
  {}^{t+\Delta t}\!\varepsilon^{e}_{ij}(0).
\end{equation}
The elastic recovery
\({}^{t+\Delta t}\!\varepsilon^{e}_{ij}(N)\) is obtained from a robust
pseudo-inverse solve of the no-flow operator, and the runtime diagnostics now
record both the coupling-update norm and the recovery residual.  The audited
regression surface shows:
\begin{itemize}
  \item the effective plastic strains remain frozen in no-flow states;
  \item the tensorial inelastic strain is updated measurably in the cracked
    unloading branch;
  \item the elastic recovery residual stays below \(10^{-8}\) in the dedicated
    no-flow regression.
\end{itemize}

When the crack topology or open/closed status changes during the same
increment, the code restarts the no-flow stress update from the new cracked
operator before applying Eq.~(26b).  This is the present approximation of the
article's inner \(m\)-loop; it is more faithful than the previous
``predictor-only'' path, but it is not yet a literal line-by-line transcription
of Table~1.

The short-run FE\textsuperscript{2} consequence was again mixed.  With the
updated Eq.~(26b) branch and the same \(45\)\,s one-step Case~5 probe, the
default secant path reached
\[
  k = 2/4,\qquad p_{\mathrm{local}} = 0.50,
\]
with residual norms
\(\|F\| = 1.9924\) and \(1.4725\) at the accepted
\(k=1/4\) and \(k=2/4\) ramp points, respectively.  The optional local
consistent-tangent override also reached \(k=2/4\) in the same wall-clock
window, with
\(\|F\| = 1.9436\) and \(1.8024\) at those two accepted ramp points.
Scientifically, this means the constitutive correction improved semantic
fidelity and local-state auditability, but it did \emph{not} by itself move the
first-step wall-clock frontier outward in the audited environment.  This
specific \(k=2/4\) frontier should now be read as an intermediate historical
measurement: after the later first-ramp continuation fix, the same Case~5
frontier moved materially farther, as documented below.

\paragraph{Tail continuation and first-predictor damping pass.}
The next stabilization step targeted exactly that late cracked tail and the
macro fixed-point transition sitting right behind it.  Two implementation
changes were introduced together:
\begin{enumerate}
  \item the persistent local solver gained an explicit \emph{tail rescue}
    policy, which restarts only the remaining segment of an already partially
    converged ramp instead of collapsing the whole solve into a flat
    \texttt{AdaptiveMinFractionReached} verdict;
  \item the iterated FE\textsuperscript{2} orchestrator began damping the
    \emph{first} micro predictor against a zero baseline, so the fixed-point
    relaxation policy acts before the second macro re-solve rather than only
    after it survives one more full iteration.
\end{enumerate}

Mathematically, if a local ramp has already accepted a fraction
\(\hat{p}\in(0,1)\) of its target and the remaining segment is re-parameterised
with \(\eta\in[0,1]\), the rescue stage solves
\begin{equation}
  \mathbf{u}(\eta)
  = \mathbf{u}(\hat{p})
    + \eta\left[\mathbf{u}(1)-\mathbf{u}(\hat{p})\right],
\end{equation}
with its own adaptive budget.  The resulting absolute minimum step length is
\begin{equation}
  \Delta p_{\min}^{\text{tail}}
  = (1-\hat{p})\,2^{-B_{\text{tail}}}\,\Delta\eta_0,
\end{equation}
which is the natural continuation scaling once most of the original segment is
already committed.  On the macro side, the first FE\textsuperscript{2}
predictor is now relaxed as
\begin{equation}
  \mathbf{r}_{\text{pred}}^{(0)}
  = \omega\,\mathbf{r}_{\mu}^{(0)}
    + (1-\omega)\,\mathbf{0},
\end{equation}
instead of injecting the full micro response at once.

\paragraph{Affine-law relaxation and macro backtracking follow-up.}
The next stabilization pass targeted exactly that macro blocker.  Two related
changes were implemented:
\begin{enumerate}
  \item \texttt{ConstantRelaxation} was redefined as the convex combination of
    two affine section laws, not as the componentwise blend of stored
    \texttt{forces} and \texttt{tangent} with a silently frozen
    \texttt{strain\_ref};
  \item \texttt{MultiscaleAnalysis} gained an explicit macro-failure
    backtracking path in operator space,
    \(\mathcal{S}^{(k)} = \alpha_k \mathcal{S}_{\mathrm{pred}}
      + (1-\alpha_k)\mathcal{S}_{\mathrm{base}}\),
    with \(\alpha_k = \beta^k\), plus a zero-correction baseline for the very
    first coupled step.
\end{enumerate}

The synthetic API regressions closed cleanly in this pass.  The new
\texttt{test\_constant\_relaxation\_preserves\_affine\_section\_law\_with\_shifted\_reference}
now verifies the affine consistency of the relaxed law explicitly, and
\texttt{test\_iterated\_two\_way\_backtracks\_macro\_predictor\_on\_failure}
verifies that a macro failure can be recovered by operator-space backtracking
under controlled conditions.

driver, a lighter executable probe,

result of the audited full-budget probe \emph{before} the zero-baseline

which exposed a real semantic gap: on the first coupled step there was no
accepted injected predictor to retreat toward, so the new backtracking path
had no admissible baseline.  The implementation was then extended to use the
zero-correction baseline on that first coupled step as well.

reduced global bisection budget was prepared for development use, but it still
did not finish within the local harness timeout.  The current state is
therefore:
\begin{itemize}
  \item the mathematical inconsistency in FE\textsuperscript{2} relaxation is
    closed and regression-tested;
  \item the macro backtracking algorithm exists, is regression-tested, and now
    covers both the ``previous predictor available'' and ``first coupled
    step'' baselines;
  \item the end-to-end Case~5 physical frontier remains honestly open until the
    new zero-baseline backtracking path is validated in a complete structural
    rerun.
\end{itemize}

\paragraph{Macro observability follow-up.}
The next execution pass then focused on making that remaining frontier
measurable rather than merely nameable.  Two additions were made:
\begin{enumerate}
  \item \texttt{NLAnalysis} now exposes the last available macro SNES
    convergence reason, iteration count, and residual norm to the FE\textsuperscript{2}
    report;
  \item \texttt{SectionHomogenizedResponse} and
    \texttt{CouplingIterationReport} now record simple diagnostics of the
    relaxed injected section operator:
    \(\lambda_{\min}(\mathbf{D}_{\mathrm{sym}})\),
    \(\lambda_{\max}(\mathbf{D}_{\mathrm{sym}})\),
    \(\operatorname{tr}(\mathbf{D})\), and the count of nonpositive diagonal
    entries.
\end{enumerate}

The purpose is not to over-interpret those diagnostics as a complete stability
theory.  The purpose is to rule out classes of explanation quickly.  After this
pass, a future Case~5 frontier row can answer questions such as:
\begin{itemize}
  \item did the macro solve fail in one violent Newton blow-up or after many
    line-search iterations?;
  \item was the injected section operator strongly indefinite at the failed
    site?;
  \item was the operator still mostly positive while the affine force intercept
    remained too aggressive?
\end{itemize}
That is a materially better scientific position than a bare
\texttt{MacroSolveFailed} label.

\paragraph{Predictor-admissibility follow-up.}
The next pass used those diagnostics operationally.  The goal was not to
``fix'' the accepted cracked local law, but to make the \emph{predictor}
injected into the next macro solve explicitly screenable.  A configurable
predictor-admissibility filter was therefore added to
\texttt{MultiscaleAnalysis}.  The current criterion is the symmetric-eigenvalue
floor
\[
  \lambda_{\min}\!\left(\mathbf{D}_{\mathrm{sym}}\right)
  \ge \lambda_{\min}^{\star},
\]
applied to the affine law that would otherwise be sent directly into the next
macro step.  If the predictor violates that floor, the algorithm blends it
toward a baseline affine law with the same geometric sequence already used by
the macro backtracking path,
\[
  \widehat{\mathcal{S}}^{(k,m)}
  = \alpha_m\,\mathcal{S}^{(k)}
  + (1-\alpha_m)\,\mathcal{S}_{\mathrm{base}},
  \qquad
  \alpha_m = \beta^m,
\]
until the floor is satisfied or the admissibility budget is exhausted.

This distinction turned out to be scientifically useful.  Macro backtracking is
only triggered \emph{after} a failed macro solve.  The new admissibility filter
acts \emph{before} the macro solve and records:
\begin{itemize}
  \item whether the filter was applied;
  \item whether the chosen admissibility floor was eventually satisfied;
  \item how many admissibility backtracking attempts were required;
  \item the accepted damping factor;
  \item the original inadmissible coupling sites.
\end{itemize}
The synthetic regression
\texttt{test\_iterated\_two\_way\_filters\_inadmissible\_predictor\_before\_macro\_resolve}
closed cleanly and verified exactly that contract on a controlled one-site
problem: a predictor with a negative minimum symmetric eigenvalue is screened,
blended toward the previous accepted baseline, and only then admitted into the
macro re-solve.

The exploratory \texttt{fe2\_crack50} profile now enables this filter with
\(\lambda_{\min}^{\star}=0\) and the same geometric factor used by macro
backtracking.  This should be read as a validation aid, not as a hidden
constitutive law change.  The accepted local response remains whatever the
micro solve delivered; only the predictor used for the next outer trial solve
is filtered.

An additional practical observation emerged while trying to reuse the cheap
\texttt{Case~5} frontier probe as the main execution artifact for this pass.
The dedicated target \texttt{fall\_n\_case5\_frontier\_probe\_test} no longer
fails at the link stage on the audited Windows/MSYS2 path.  The recovery was
purely architectural and build-level:
\begin{itemize}
  \item the heavy base-shear extraction path was moved behind the concrete
    wrapper \texttt{extract\_base\_shear\_x\_struct\_model(...)} in
    \texttt{TableCyclicValidationSupport.cpp}, so the probe target no longer
    instantiates that template-heavy path directly;
  \item PCH was disabled for the probe target, which avoided a second layer of
    target-specific coupling on the Windows/MSYS2 toolchain;
  \item \texttt{SubmodelMaterialFactoryDefaults.hh} received the explicit
    material include it had been inheriting only accidentally from the PCH.
\end{itemize}
After those corrections, the target builds again sequentially in the audited
environment.  The honest remaining limitation is now runtime, not link/build:
the cheap probe is still wall-clock limited in the local harness, so it is not
yet a stable CI-speed oracle for the full structural Case~5 frontier.

\paragraph{Macro step-cutting follow-up.}
Once the probe could be rebuilt again, the next execution pass targeted the
macro solve itself rather than the micro constitutive path.  The practical
question was: if the cracked micro operators are already acceptable, does the
first macro FE\textsuperscript{2} re-solve still fail simply because the
solver tries to traverse too much of the global path in a single increment?

The implemented answer is a target-preserving macro cutback layer.  Let the
committed macro control parameter at the start of the FE\textsuperscript{2}
step be \(p_n\), let the nominal solver increment be \(\Delta p_n\), and let
the intended target remain
\begin{equation}
  p_{n+1}^{\star} = p_n + \Delta p_n.
\end{equation}
If the macro solver exposes runtime increment control, the FE\textsuperscript{2}
orchestrator may now retry the same target with a reduced internal increment
\begin{equation}
  \Delta p_n^{(k)} = \beta^k \Delta p_n,
  \qquad
  k = 1,\dots,N_{\mathrm{cut}},
\end{equation}
while solving
\begin{equation}
  p_n \;\xrightarrow[\Delta p_n^{(k)}]{\texttt{step\_to}}\; p_{n+1}^{\star}.
\end{equation}
The important detail is that the FE\textsuperscript{2} protocol target is
\emph{not} changed; only the internal macro continuation path is refined.  This
makes the method a continuation aid for the validation campaign, not an
unreported change of loading program.

The dedicated synthetic regression
\texttt{test\_iterated\_two\_way\_recovers\_macro\_failure\_with\_step\_cutback}
closed cleanly on the typed FE\textsuperscript{2} API harness.  In that test,
the first macro trial fails when the solver increment is larger than \(0.5\),
but the same FE\textsuperscript{2} target converges once the orchestrator
retries it with a \(0.5\) cutback factor.  The report now records:
\begin{itemize}
  \item whether macro step cutting was attempted;
  \item whether it succeeded;
  \item the number of cutback retries;
  \item the nominal increment of the FE\textsuperscript{2} step;
  \item the accepted reduced increment;
  \item the accepted reduction factor.
\end{itemize}
The exploratory \texttt{fe2\_crack50} profile now enables a small cutback
budget in addition to predictor admissibility filtering and operator-space
macro backtracking.  The full structural Case~5 rerun remains runtime-limited
in the audited harness, so the present claim is deliberately narrow: the
continuation capability exists, is regression-tested, is wired through the
validation API and recorder CSVs, and is available for the next physical
validation pass.

The verification evidence for this pass was:
\begin{itemize}
  \item \texttt{fall\_n\_multiscale\_api\_test.exe}: green, including the new
    cutback regression and the previously added affine-relaxation,
    macro-backtracking, and predictor-admissibility checks;
  \item \texttt{fall\_n\_cyclic\_validation\_api\_test.exe}: green, verifying
    that the exploratory \texttt{fe2\_crack50} profile now wires predictor
    admissibility, macro step cutting, and macro backtracking coherently;
  \item \texttt{fall\_n\_case5\_frontier\_probe\_test}: build green again after
    the target cleanup, but runtime-limited in the audited harness.
\end{itemize}
This combination is scientifically useful even though it does not yet close the
full Case~5 path: the first two artifacts validate the algorithmic contract,
and the third confirms that the remaining frontier is a runtime/methodology
question rather than a target-build defect.

\paragraph{Workflow hardening follow-up.}
This same pass also closed a practical CI blocker.  The Windows/MSYS2 GitHub
workflow built the regression executables successfully under the UCRT64 toolchain,

small operationally, but it matters because a broken gate would otherwise make
the FE\textsuperscript{2} stabilization claims non-reproducible on the public
CI surface.  It also records a more honest interpretation of the runner failure:
the audited local evidence showed that \texttt{fall\_n\_evolver\_advanced\_test}
itself was sound, so the public failure was treated as an environment-closure
problem rather than as a constitutive regression.

\section{Follow-up: Material-Module Segregation of the Reference Local Law}
\label{sec:ms-stab-material-segregation}

\subsection{Motivation}

The audit showed that the FE\textsuperscript{2} core had already been decoupled
from driver logic, but the continuum local-model path still carried one
architectural shortcut: the multiscale/reconstruction layer still owned the
concrete default types \texttt{KoBatheConcreteMaterialFactory} and
\texttt{MenegottoPintoRebarFactory}.  That was acceptable for a reinforced
concrete validation driver, but it was not acceptable as the longer-term
publication architecture because it implicitly suggested that the local
operator was ``the Ko--Bathe submodel'' instead of ``a local model satisfying
the FE\textsuperscript{2} contract''.

The scientific consequence is not merely aesthetic.  If the local operator is
conceptually frozen to one constitutive family, it becomes harder to insert
future alternatives such as enriched crack local models, XFEM-like
formulations, or broken-test-space / discontinuous Petrov--Galerkin local
solvers.  The stabilization cycle therefore required a sharper dependency
boundary.

\subsection{Implemented change}

The change was deliberately conservative in runtime terms:
\begin{itemize}
  \item abstract factory contracts were moved to
    \texttt{src/materials/SubmodelMaterialFactory.hh};
  \item the current reference defaults
    (\texttt{KoBatheConcreteMaterialFactory} and
    \texttt{MenegottoPintoRebarFactory}) were moved to
    \texttt{src/materials/SubmodelMaterialFactoryDefaults.hh};
  \item \texttt{src/reconstruction/MaterialFactory.hh} was reduced to a
    forwarding compatibility shim;
  \item \texttt{NonlinearSubModelEvolver} now constructs its default materials
    through \texttt{make\_default\_submodel\_concrete\_factory(...)} and
    \texttt{make\_default\_submodel\_rebar\_factory(...)}, so the
    reconstruction layer no longer names Ko--Bathe/Menegotto directly;
  \item \texttt{SubModelSolver} was updated to use the same abstract factory
    boundary and now accepts an injected continuum-material factory, which
    closes a remaining legacy coupling point;
  \item the cyclic validation factory helper was updated to depend on the new
    materials-module defaults explicitly rather than on the reconstruction shim.
\end{itemize}

No new virtual dispatch was introduced in the micro hot path.  Factory
dispatch occurs only when the local model is instantiated or rebuilt, whereas
the Gauss-point update, tangent assembly, sparse condensation, and FE\textsuperscript{2}
fixed-point loop remain template-based exactly as before.  In other words, the
change improves dependency structure without paying for runtime polymorphism in
the critical inner solve.

\subsection{Architectural reading}

After the change, the conceptual layering is:
\begin{center}
\begin{tabular}{c}
\texttt{MultiscaleAnalysis / BeamMacroBridge / MultiscaleModel} \\
\(\downarrow\) local-model contract \\
\texttt{LocalModelAdapter} \\
\(\downarrow\) persistent continuum implementation \\
\texttt{NonlinearSubModelEvolver} \\
\(\downarrow\) material construction contracts \\
\texttt{ConcreteMaterialFactory / RebarMaterialFactory} \\
\(\downarrow\) reference implementations (materials module) \\
\texttt{KoBatheConcreteMaterialFactory / MenegottoPintoRebarFactory}
\end{tabular}
\end{center}

This makes the FE\textsuperscript{2} kernel materially agnostic in the precise
software-engineering sense intended by the publication plan: the orchestration
layer depends on a local operator interface, and the current continuum local
implementation depends on material-construction contracts.  The Ko--Bathe law
is therefore one valid realization of the current local continuum path, not a
hard-coded property of the multiscale core.

\subsection{Verification}

The refactor was verified with:
\begin{itemize}
  \item \texttt{fall\_n\_sub\_model\_solver\_test}, extended with an injected
    purely elastic continuum factory to prove that \texttt{SubModelSolver}
    no longer requires a Ko--Bathe material to function;
  \item \texttt{fall\_n\_evolver\_advanced\_test}, which continues to exercise
    the Ko--Bathe reference factory path and therefore checks that the move did
    not regress the current nonlinear local benchmark surface;
  \item \texttt{fall\_n\_cyclic\_validation\_api\_test}, which checks that the
    validation driver still builds the intended reference submodel factories
    after the module-boundary change.
\end{itemize}

\subsection{Residual debt after the segregation}

The first residual debt identified at the beginning of the pass was that
\texttt{SubModelSolver::solve\_reinforced()} still estimated the penalty-coupling
scale from the reinforced-concrete proxy
\(E_c \approx 4700\sqrt{f_c}\).  In the follow-up implementation of the same
cycle, that assumption was moved behind an explicit value-type policy,
\texttt{EmbeddedLinePenaltyCouplingConfig}, with two modes:
\begin{equation}
  \alpha =
  \begin{cases}
    \alpha_{\mathrm{explicit}}, &
    \texttt{EmbeddedLinePenaltyMode::ExplicitAlpha},\\[4pt]
    \gamma E_c, \quad E_c \approx 4700\sqrt{f_c}, &
    \texttt{EmbeddedLinePenaltyMode::ReferenceConcreteHeuristic}.
  \end{cases}
  \label{eq:ms-stab-embedded-line-penalty-policy}
\end{equation}
This does not make the legacy solver fully material-agnostic yet, but it does
change the architectural semantics materially: the reinforced-concrete penalty
estimate is now a default policy choice rather than a hidden constant baked
into the only available reinforced local solve path.

The same pass also closed a concrete CI regression in the example drivers:
\texttt{main\_lshaped\_multiscale.cpp},
\texttt{main\_lshaped\_multiscale\_16storey.cpp}, and
\texttt{main\_table\_multiscale.cpp} still wrote
\texttt{CrackSummary::max\_damage}, even though the stabilized crack-summary
semantics had already moved to the pair
\texttt{damage\_scalar\_available} / \texttt{max\_damage\_scalar}.  The fix was
not merely syntactic.  The CSV outputs of those examples now follow the same
honest semantics as the cyclic validation harness:
\begin{equation}
  \texttt{damage\_scalar\_available} = 0
  \quad\Longrightarrow\quad
  \texttt{max\_damage\_scalar} = \mathrm{NaN},
  \label{eq:ms-stab-example-damage-csv-semantics}
\end{equation}
instead of silently reporting zero damage when the constitutive snapshot does
not expose a scalar damage channel.

\section{Failed-Step Recorder Hardening and Fast Case~5 Frontier Audit}
\label{sec:ms-stab-failed-step-frontier-audit}

\subsection{Motivation}

The previous Case~5 frontier pass had already isolated a useful cheap command,
but it still suffered from an observability defect.  When the
FE\textsuperscript{2} step aborted, only
\texttt{solver\_diagnostics.csv} received a row.  The higher-level
\texttt{global\_history.csv} and \texttt{crack\_evolution.csv} files remained
empty because they were written only on accepted FE\textsuperscript{2} steps.

That was not merely inconvenient.  It meant the validation campaign could
locate a failure through solver metadata, but could not preserve the physical
interpretation of that failure in the same recorder family used for accepted
steps.  The stabilization goal of this pass was therefore twofold:
\begin{itemize}
  \item preserve failed-step evidence without pretending that a rolled-back
    step became accepted physics;
  \item make the first cracked Case~5 point reproducible through a named,
    fast audit profile.
\end{itemize}

\subsection{Implementation}

The implementation introduced two complementary changes.

\paragraph{Last-attempt crack summary.}
\texttt{NonlinearSubModelEvolver} now stores a
\texttt{last\_attempted\_crack\_summary} in addition to the committed
\texttt{crack\_summary}.  The committed summary is still updated only through
\texttt{end\_of\_step()}, while the attempted summary is refreshed after every
\texttt{solve\_step()} through the lightweight crack-diagnostics path with
\texttt{retain\_detail = false}.  This preserves the last attempted local
constitutive observables across FE\textsuperscript{2} rollback without
changing the semantics of the committed VTK-oriented crack output.

\paragraph{Failed-step recorder contract.}
The FE\textsuperscript{2} recorders now append rows even when the coupled step
fails.  The new CSV contract adds:
\begin{itemize}
  \item \texttt{accepted};
  \item \texttt{termination\_reason};
  \item \texttt{rollback\_performed};
  \item \texttt{total\_active\_crack\_history\_points};
  \item \texttt{max\_num\_cracks\_at\_point}.
\end{itemize}
Accepted structural quantities that would be undefined after rollback are
written as \texttt{NaN}; the failed-step rows do \emph{not} attempt to
reconstruct a false accepted base shear or structural damage from the restored
checkpoint.

\subsection{Fast frontier-audit profile}

The validation API now exposes a dedicated profile
\texttt{fe2\_frontier\_audit}, wired through
\texttt{ValidationExecutionProfile::FE2FrontierAudit}.  Its purpose is to
reach the first failed cracked Case~5 point with lower runtime cost than the
production-like \texttt{fe2\_crack50} budget.  The audited command was:
\begin{verbatim}
build/fall_n_table_cyclic_validation.exe
  --case 5
  --protocol extended50
  --fe2-profile frontier
  --max-steps 1
  --global-output-interval 0
  --submodel-output-interval 0
\end{verbatim}
The run completed successfully as a process in about \(698\)\,s and produced a
single failed-step row in all three FE\textsuperscript{2} recorder families.

\subsection{Observed result}

The first step now fails reproducibly as
\begin{equation}
  \texttt{termination\_reason} = \texttt{MicroSolveFailed},
  \qquad
  \texttt{failed\_submodels} = 2,
  \qquad
  \texttt{rollback\_performed} = 1.
\end{equation}
The failed coupling sites are the first Gauss points of elements \(0\) and
\(2\), both at \(\xi=-0.57735\).

The recorded observables are split exactly as intended:
\begin{equation}
  \texttt{accepted} = 0,
  \qquad
  \texttt{base\_shear\_MN} = \mathrm{NaN},
  \qquad
  \texttt{peak\_damage} = \mathrm{NaN},
\end{equation}
while the micro-side crack metrics remain available:
\begin{equation}
  \texttt{total\_cracked\_gps} = 0,
  \quad
  \texttt{total\_cracks} = 0,
  \quad
  \texttt{max\_opening} = 0,
  \quad
  \texttt{total\_active\_crack\_history\_points} = 306,
  \quad
  \texttt{max\_num\_cracks\_at\_point} = 3.
\end{equation}
This is an important scientific clarification.  The first failed cracked point
is no longer ``missing data'' and it is no longer misread as ``no cracking''.
Instead, the record now says exactly what happened: crack history is already
active in many material points, but no open-crack observable survives the
current recorder criterion at that failed FE\textsuperscript{2} target.

\subsection{Interpretation}

After this pass the Case~5 validation frontier is best understood as a
two-level surface:
\begin{itemize}
  \item under the cheap \texttt{fe2\_frontier\_audit} budget, the first failed
    point is micro-limited and localized at two coupling sites;
  \item under the larger \texttt{fe2\_crack50} budget, the same first cracked
    point is already macro-dominated after all four local columns converge.
\end{itemize}
This split is more valuable than a single failure label because it guides the
next validation work precisely: the local late cracked tail and the macro
post-peak continuation problem can now be studied separately.

\section{Closure of the Stabilization Cycle}
\label{sec:ms-stab-close}

The multiscale stabilization cycle v1 closed the most important gap between
``public API redesign'' and ``release-worthy behaviour'':
\begin{enumerate}
  \item the local tangent path now exposes its own uncertainty instead of
    silently zeroing columns;
  \item incomplete local section operators are now classified explicitly as
    invalid rather than being treated as weak-but-usable responses;
  \item the FE\textsuperscript{2} protocol reports why it terminated and
    whether it rolled back;
  \item the physical path now has deterministic axial, flexural, shear, and
    torsional regression coverage;
  \item the release gate exists both as a local script and as a repository
    workflow;
  \item the debt that remains is explicit, bounded, and technically
    understandable.
\end{enumerate}
